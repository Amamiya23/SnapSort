# 筛选界面显示照片名称与曝光信息

## Goal

在照片筛选界面为当前照片显示更有判断价值的元信息：照片文件名，以及 EXIF 中可读取的曝光三要素（光圈、快门、ISO）。信息呈现要服务快速筛选，保持 SnapSort 现有的克制、照片优先、低干扰审美，避免把界面变成文件管理器或堆满标签的图库。

用户价值：

- 摄影师能在筛选时快速确认当前照片身份，尤其是连拍/相近构图中通过文件名定位照片。
- 用户能参考曝光参数判断画面质量和取舍原因。
- EXIF 缺失或读取失败时，筛选流程不被打断。

## Confirmed Facts

- `PhotoEntity` 已持久化 `fileName`、`baseName`、`extension`，筛选界面目前没有显示文件名。
- `FolderScanner` 当前使用 `ExifInterface` 读取 `TAG_DATETIME_ORIGINAL`，未读取光圈、快门、ISO。
- 筛选界面 `PhotoSelectionScreen` 顶部目前显示分组进度、拍摄时间和 RAW 有无。
- `PhotoSelectionViewModel` 将 `PhotoEntity` 映射为 `PhotoSelectionItem`，适合作为 UI 显示字段的边界。
- 新增曝光字段需要触及扫描、核心扫描模型、Room 实体、Repository 映射、ViewModel 和 Compose UI。
- Room 当前版本为 1，且 `exportSchema = true`。新增持久字段需要版本/迁移或明确兼容策略。

## Requirements

- 在照片筛选界面显示当前照片的 JPG 文件名。
- 显示 EXIF 曝光三要素：
  - 光圈，例如 `f/2.8`
  - 快门，例如 `1/500` 或 `0.5s`
  - ISO，例如 `ISO 400`
- 三要素字段均为可空；缺失单项时只省略该项，不显示错误、占位符或技术性空值。
- 当三要素全部缺失时，界面仍显示文件名、拍摄时间和 RAW 状态，且不出现空白布局或异常。
- 文件名和曝光信息应以低干扰方式呈现，不遮挡主要照片判断区域，不削弱标记删除/保留等核心操作。
- 信息常驻在筛选页顶部信息区；文件名与 RAW 状态同占第二行；光圈、快门、ISO 在下一行紧凑展示；选图界面不显示拍摄时间。
- 显示内容应支持明暗主题、动态颜色、系统字体缩放和横竖屏/窄屏布局。
- EXIF 读取应留在扫描层或数据层，Compose UI 不直接访问 `ContentResolver` 或 EXIF。
- 扫描时读取 EXIF 失败不得中断扫描；曝光字段缺失应作为正常情况处理。
- 从旧版本升级后，已有最近任务的曝光字段为空时，进入筛选界面应尝试后台读取当前照片 EXIF 并回填，不要求用户重新扫描才能看到曝光信息。
- 现有删除标记、RAW 匹配、分组顺序、打开动画和手势反馈行为不得回退。

## Acceptance Criteria

- [ ] 筛选界面当前照片区域能看到 JPG 文件名。
- [ ] 有完整 EXIF 的 JPG 扫描后，筛选界面显示光圈、快门和 ISO。
- [ ] 缺少部分 EXIF 时，界面仅显示可用项，并保持视觉平衡。
- [ ] 缺少全部曝光 EXIF 时，界面不显示占位符，不崩溃，文件名/RAW 信息仍可见。
- [ ] 旧任务数据库行曝光字段为空时，打开筛选界面可后台补读当前照片 EXIF；读取成功后界面自动出现曝光信息。
- [ ] 文件名过长时不会与返回按钮、操作按钮或照片内容产生明显重叠；文本可截断或换行但布局稳定。
- [ ] 新增 Room 字段有明确迁移/版本处理，旧数据库升级后应用可启动并重新扫描使用新字段。
- [ ] 扫描、Repository、ViewModel、UI 的数据流边界符合 `.trellis/spec/android` 指南。
- [ ] `:app:testDebugUnitTest` 与 `:app:compileDebugKotlin` 通过。
